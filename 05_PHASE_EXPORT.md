# stayd-banda: Phase 5 - Export and Evidence Packs

Upload 00_SHARED_CONTEXT.md alongside this file.

## What This Session Builds

1. ZIP download of all photos per turnover
2. ZIP download of flagged photos only
3. PDF evidence report (the feature that directly supports Airbnb damage claims)
4. Bulk property download with date range

## Assumes Phases 1-4 Are Complete

- Turnovers with photos exist
- Photo viewer and search working
- Presigned download URLs working

## Dependencies to Install

```bash
npm install archiver        # ZIP file generation
npm install @react-pdf/renderer  # PDF generation (server-side)
# OR
npm install puppeteer-core  # Alternative PDF approach via HTML rendering
```

Recommendation: use @react-pdf/renderer. It runs in Node.js without a browser dependency, which matters on Vercel serverless. Puppeteer requires a Chromium binary that adds cold start time and may exceed Vercel function size limits.

If @react-pdf/renderer proves too limited for the layout, fall back to generating HTML and converting with a lightweight tool. But try @react-pdf/renderer first.

## Step by Step

### 1. ZIP Download Per Turnover

**API route: GET /api/export/turnover/[id]**

Query params:
- flagged_only=true (optional, filters to flagged photos only)

Flow:
1. Verify user has access to the turnover's property
2. Fetch all photos for the turnover (or flagged only)
3. For each photo, generate a presigned download URL for the original image
4. Fetch each original image from R2
5. Stream into a ZIP archive using archiver

ZIP structure:
```
turnover_[checkout-date]_to_[checkin-date]/
  post-checkout/
    kitchen/
      photo_001.jpg
      photo_002.jpg
    bedroom-1/
      photo_003.jpg
    general/
      photo_004.jpg
  pre-checkin/
    kitchen/
      photo_005.jpg
    bedroom-1/
      photo_006.jpg
```

Filenames: use sequential numbering with area prefix for readability. Preserve original file extension.

Response: stream the ZIP directly to the client with headers:
```
Content-Type: application/zip
Content-Disposition: attachment; filename="turnover_2026-03-01_to_2026-03-03_[property-name].zip"
```

**Performance note:** For turnovers with many photos (50+), this could be slow on a serverless function. The Vercel Hobby (free) tier has a 10 second function timeout which will not be enough. Upgrade to Vercel Pro ($20/month) before building this phase. Pro gives 60 second timeout.

In vercel.json, set the function timeout:
```json
{
  "functions": {
    "src/app/api/export/**/*.ts": {
      "maxDuration": 60
    }
  }
}
```

For v1, 60 seconds is sufficient. If it proves too tight later, consider generating exports in the background and emailing a download link.

### 2. PDF Evidence Report

**API route: GET /api/export/turnover/[id]/report**

This is the feature that makes banda directly useful for Airbnb Resolution Centre claims.

**PDF layout:**

Page 1: Cover
- "Property Condition Report"
- Property name and address
- Turnover dates: checkout [date], checkin [date]
- Departing guest reference (if provided)
- Generated by: banda by stayd
- Generated at: [timestamp]
- Total photos: [count]
- Flagged items: [count]

Pages 2+: One section per area

For each area:
- Area name as section header
- Two-column layout:
  - Left column: "After guest left" (post_checkout)
  - Right column: "Before next guest" (pre_checkin)
- Photos displayed at a reasonable size (roughly 300px wide in the PDF)
- Below each photo:
  - Captured: [EXIF timestamp or "Not available"]
  - Uploaded: [server timestamp]
  - Device: [make model or "Not available"]
  - GPS: [lat, lon or "Not available"]
  - Uploaded by: [user name]
- If a photo is flagged:
  - Red border around the image
  - "DAMAGE FLAGGED" label
  - Damage note text below

If an area has photos in only one set (e.g. post_checkout but not pre_checkin), show the photos in their column and leave the other column with "No photos for this set."

**Flagged items summary page (after all areas):**

If any photos are flagged:
- "Damage Summary" page
- List each flagged photo with: area name, photo set, capture timestamp, damage note, thumbnail
- This gives a quick overview of all damage for the claim

**Footer on every page:**
- "Generated by banda (banda.stayd-tools.com). All photographs are original, unmodified uploads with server-verified timestamps. No AI generated or AI enhanced content."
- Page number

**Implementation with @react-pdf/renderer:**

```jsx
// Pseudocode structure
import { Document, Page, View, Text, Image, StyleSheet } from '@react-pdf/renderer';

const EvidenceReport = ({ property, turnover, areas, photos }) => (
  <Document>
    <Page style={styles.page}>
      {/* Cover page */}
      <View style={styles.cover}>
        <Text style={styles.title}>Property Condition Report</Text>
        <Text>{property.name}</Text>
        <Text>{property.address}</Text>
        {/* etc */}
      </View>
    </Page>
    
    {areas.map(area => (
      <Page key={area.id} style={styles.page}>
        <Text style={styles.areaHeader}>{area.name}</Text>
        <View style={styles.columns}>
          <View style={styles.column}>
            <Text style={styles.columnHeader}>After guest left</Text>
            {/* post_checkout photos for this area */}
          </View>
          <View style={styles.column}>
            <Text style={styles.columnHeader}>Before next guest</Text>
            {/* pre_checkin photos for this area */}
          </View>
        </View>
      </Page>
    ))}
    
    {/* Damage summary page if flagged photos exist */}
  </Document>
);
```

To render on the server:
```js
import { renderToBuffer } from '@react-pdf/renderer';
const buffer = await renderToBuffer(<EvidenceReport {...data} />);
```

Return with headers:
```
Content-Type: application/pdf
Content-Disposition: attachment; filename="evidence_report_[property]_[checkout-date].pdf"
```

**Image handling in PDF:**
- Fetch each photo from R2 (use thumbnails for the report, not originals, to keep PDF size manageable)
- Convert to base64 data URLs for embedding in the PDF
- If a turnover has 100+ photos, the PDF could be very large. Cap at thumbnail resolution

### 3. Bulk Property Download

**API route: GET /api/export/property/[id]**

Query params:
- from (date, required): start of date range (checkout_date >=)
- to (date, required): end of date range (checkout_date <=)

Flow:
1. Verify user is owner with access to the property
2. Fetch all turnovers for the property within the date range
3. Fetch all photos for those turnovers
4. Build ZIP with structure:

```
[property-name]/
  turnover_2026-03-01_to_2026-03-03/
    post-checkout/
      kitchen/
        photo_001.jpg
      ...
    pre-checkin/
      ...
  turnover_2026-03-05_to_2026-03-08/
    ...
```

Same streaming approach as single turnover ZIP.

This will likely hit Vercel timeout for properties with many turnovers. Acceptable for v1 with the 60 second limit. Add a warning in the UI: "Large date ranges may take a moment to generate."

### 4. UI Integration

**Turnover detail page (/turnovers/[id]) buttons:**

Replace the disabled placeholders from Phase 4 with working buttons:

- "Download all photos" button: triggers GET /api/export/turnover/[id], downloads ZIP
- "Download flagged only" button: triggers GET /api/export/turnover/[id]?flagged_only=true
- "Download evidence report" button: triggers GET /api/export/turnover/[id]/report, downloads PDF

Show a loading spinner while the file generates. Use a client-side approach:
```js
const downloadFile = async (url, filename) => {
  const response = await fetch(url);
  const blob = await response.blob();
  const link = document.createElement('a');
  link.href = URL.createObjectURL(blob);
  link.download = filename;
  link.click();
  URL.revokeObjectURL(link.href);
};
```

Add "Export" section or button:
- Date range picker (from / to)
- "Download all turnovers" button: triggers GET /api/export/property/[id]?from=...&to=...
- Warning text: "Large date ranges may take a while to generate"
